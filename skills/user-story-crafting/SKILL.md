# user-story-crafting — 场景化用户故事生成

通过询问式对话，从用户自然语言（可能含工具假设）中反向拆解，逐步产出 **1 个场景化用户故事 + 业务验收标准**。不写交互验收标准（原型之后的事）。

## 流程

### 步骤 A：需求反向拆解 [自由度：低]
**适用场景**：用户说出带功能假设的描述（"我要做工单系统"、"要能拖拽看板"）。
→ **此时加载 [references/guide-step-a.md](references/guide-step-a.md)**

从描述中拆出 4 项：**业务场景、角色、痛点、原始诉求**。
展示格式：
```
**业务场景**：[推断] — 对吗？
**涉及角色**：[推断] — 符合实际？
**痛点猜想**：[推断] — 还有更关键的？
**原始诉求**：[不含工具假设] — 核心目标是这个？
```
**核心规则：每项必须经用户确认/纠正后才能进入下一步。**

### 步骤 0：前置确认 [自由度：低]
- 确认场景范围
- **多故事分支**：同一场景多角色 → 共享步骤 2，各角色独立走 3→4→5；不同场景 → 逐个走完

### 步骤 1：询问角色 [自由度：高]
如果步骤 A 已有角色 → 直接用，跳至步骤 2。信息不足时补充："还有哪些参与者？"

### 步骤 2：现状工作流 [自由度：低]
→ **此时加载 [references/guide-step-2.md](references/guide-step-2.md)**

**三问顺序（严格）**：
1. 当前方式 → 2. 痛点 → 3. 原始诉求
→ 提炼为当前方式 + 痛点 + 原始诉求，展示确认后进入步骤 3

### 步骤 3：AIS 用户故事 [自由度：低]
→ **此时加载 [references/guide-step-34.md](references/guide-step-34.md)**

从原始诉求提炼，禁止引工具名/参数。逐词检查禁止词表。

### 步骤 4：业务验收标准 [自由度：低]
提问："满足什么条件算做成？前置条件？什么情况算失败？"
- 具体规则 → GWT | 模糊 → 自然语言
- **只写业务层，不写交互层**

### 步骤 5：交付 [自由度：低]
按模板组装（模板见 [references/guide-step-34.md](references/guide-step-34.md) 末尾）。
展示完整内容，确认后交付。

## 约束规则 [自由度：无]

| 维度 | 规则 |
|------|------|
| **防止臆想** | 拆解/提炼内容逐项展示确认，不臆想填补 |
| **后视偏差** | 从"没工具时怎么办"开始，现状工作流未完成禁止生成故事 |
| **角色** | ❌ 系统管理员/审核员/运营 | ✅ 真实业务身份 |
| **Want** | ❌ 配置/管理/设置/编辑/维护 | ✅ 提交/查看/通知/审批/分配/追踪 |
| **So that** | ❌ 系统效果 | ✅ 人的收益 |
| **验收标准** | ❌ 交互层 | ✅ 业务层 |

### 验收标准反模式示例

| ❌ 不合格（混入交互细节） | ✅ 合格（纯业务层） |
|---------------------------|-------------------|
| "点击提交按钮后，24小时内收到审批结果通知" | "提交申请后，24小时内完成审批并通知申请人" |
| "在详情页显示审批进度，支持按列排序" | "申请人可随时查看审批进度" |
| "审批通过后页面跳转到列表页" | "审批完成后，申请人收到结果通知" |
| "提交按钮置灰，审批后才能再次提交" | "同一申请不可重复提交" |
| "列表页筛选出待办工单" | "工单可按状态筛选" |

**禁止词表**：配置、管理、设置、编辑、维护、系统自动、数据同步、提高效率、优化体验

**交互检测词表**（验收标准中出现以下词即判定为交互层）：按钮、弹窗、页面、跳转、点击、链接、输入框、下拉框、列表页、详情页、标签页、滚动、拖拽（仅指界面操作时）、置灰、高亮、图标、Tab、选项卡、面包屑

### 验收标准完成的额外检查

- [ ] 验收标准中不包含界面元素、组件、页面路径、按钮操作、跳转等交互描述
- [ ] 验收标准中不出现交互检测词表中的词汇
- [ ] 验收标准描述的是业务规则和结果，不是用户怎么操作界面

## 失败回退
- 不认可 → 调整重试（最多 3 轮），超限汇报阻塞

## 完成标准
- [ ] 产出物按模板格式完整，用户逐项确认，无约束违规
