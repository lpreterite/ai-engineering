# 隔离上下文规则

> SKILL.md 步骤 2 时加载。Tester 执行测试时必须遵守的隔离规则。

## 核心原则

Tester **不看** Developer 的代码和测试用例，仅基于验收标准独立设计测试方案。目的是避免确认偏误（confirmation bias）。

## 允许访问

- Issue 正文（含验收标准、相关文档引用）
- PRD / Tech Spec / Design Spec
- 已部署的测试环境
- 项目文档中的测试策略说明

## 禁止访问

- Developer 的代码变更（diff）
- Developer 写的单元测试/集成测试
- 开发分支的 commit 记录
- Developer 的自测过程

## Bug 严重程度分级

| 级别 | 定义 | 示例 |
|------|------|------|
| **P0 阻塞级** | 核心功能不可用，无绕行方案 | 登录失败、数据错误、主流程断裂 |
| **P1 严重级** | 功能异常，但有临时绕行方案 | 某个子功能不可用，有替代路径 |
| **P2 一般级** | 非核心异常或体验问题 | UI 错位、文案错误、非关键链路超时 |

## Tester 工作流

```
接收移交
    │
    ▼
独立设计测试用例（基于验收标准）
    │
    ▼
执行测试
    │
    ├── 全部通过 → Issue 标记 verified
    │
    └── 发现问题 → 记录 Bug → Orchestrator 创建 Bug Issue
                     → Issue 设回 in_progress
                     → Developer 修复
```

## 测试证据要求

每个验收标准至少包含以下证据之一：
- 控制台/日志截图
- API 响应截图
- 自动化测试结果输出
- 人工验证记录

## Bug Report 模板

```
## Bug Report

**BUG-ID**: BUG-001
**环境**: staging / production / local
**步骤**:
1. 打开页面 X
2. 执行操作 Y
3. 观察到结果 Z

**预期行为**:
[根据验收标准描述]

**实际行为**:
[当前观察到的行为]

**证据**:
[截图/日志/API 响应]

**严重程度**: P1
**关联 Issue**: #N（父 Issue）
```
