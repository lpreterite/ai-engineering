# 冲突解决

> SKILL.md 步骤 2 时加载。5 种冲突场景的处理方式。

## 5 种冲突场景

| 场景 | 上游 | 下游 | 处理策略 |
|------|------|------|----------|
| **1. 仅上游更新** | 有变更 | 未定制 | 直接覆盖，保留 MANIFEST 版本号 |
| **2. 仅下游定制** | 无变更 | 有定制 | 跳过，保持下游版本 |
| **3. 两者更新（无冲突）** | 有变更 | 有定制但区域不重叠 | 三路合并自动合并 |
| **4. 两者更新（有冲突）** | 有变更 | 有定制且区域重叠 | 三路合并产生冲突标记，人工裁决 |
| **5. 新增文件** | 新增 | 不存在 | 直接复制，标记为 `new` |

## DOWNSTREAM-BEGIN/END 标记

在下游修改规范文件时，用标记包围定制区域：

```markdown
<!-- DOWNSTREAM-BEGIN: 本项目增加了 Code Review 阶段 -->
### 2.5 Code Review
本项目要求所有 PR 必须经过 Review 后方可合并。
<!-- DOWNSTREAM-END -->
```

同步时 AI 自动识别标记区间，避免覆盖。

## 冲突标记处理

三路合并产生 `<<<<<<<` / `=======` / `>>>>>>>` 标记时：
1. 读取冲突区域上下文
2. 判断是定制内容冲突还是上游变更冲突
3. 若无法自动裁决，标记为待人工处理
4. 在变更摘要中列出所有未解决冲突的文件

## 最佳实践

- 定制内容放在独立小节（新增章节而非修改上游章节）
- HEADER 元信息区块尽量不改
- 在 MANIFEST.json 中为定制文件记录 `customized_reason`
